# Стиль текста и общения (TasteCompass и интерфейсы этого проекта)

Пользовательский текст **не должен читаться как строка кода, тикет или README для разработчика**. Пишем так, чтобы **человек, который ничего не знает о продукте**, понял с первого прочтения. Отзыв: [RESPONDENT-02-ELEONORA.md](./feedback/RESPONDENT-02-ELEONORA.md).

Глобальное правило Cursor (все проекты): `~/.cursor/rules/human-copy-full-thoughts.mdc`.

---

## Описания на сайт (зафиксированный чеклист)

Так делаем тексты на сайте и в UI этого продукта:

- **Живая разговорная норма** (как объясняешь другу, не как инструкция к прибору).
- **Цельные мысли** (одна мысль = связное предложение или абзац, не рубленые ярлыки).
- **Прямой смысл с первого чтения** (человек с нуля понимает, зачем ему это).
- **Без сокращений и жаргона**, если это не общепринятое слово в теме.
- **Простой понятный язык** (без канцелярита и «умных» слов ради красоты).
- **Цельность:** не обрывки вроде отдельной строки «Зачем тебе.» без нормального предложения.
- **Связность:** части фразы соединять союзами и запятыми, а не рвать посередине (длинное тире U+2014 в UI не использовать, см. раздел «Главное» ниже).
- **Адресат:** писать так, будто читатель не в курсе, что мы делаем внутри команды.
- **Без лишних отписок:** не тащить «без регистрации», если это не помогает человеку.

### Ясность для уха (обязательно)

Не опираться на **укороченное второе сказуемое** после запятой («учебный пример, не афиша») без явной связи. Для противопоставления использовать **«…, а не …»** или **две фразы**, или союз **а** / **и** так, чтобы было слышно с первого чтения. По умолчанию выбирать формулировку, которая **для носителя яснее**, а не самую короткую за счёт обрыва.

## Главное

1. **Цельные предложения и мысли.** Не рубить текст на обрывки и не цеплять куски знаком «сейчас объясню потом». Если нужно две части, связывай **и**, запятой или точкой, а не тире между половинами фразы.
2. **Заголовки тоже должны быть нормальными фразами.** Не «Зачем тебе.» одно слово с точкой, а например **«Зачем тебе это»** или вообще без ярлыка, одним абзацем.
3. **Польза для человека**, не внутренняя цель команды. Не «проверить идею», не стек «без X, без Y, только Z», если это не помогает читателю (часто лучше убрать совсем).
4. **Символ «длинное тире» (Unicode U+2014, визуально длинная черта между частями предложения) в пользовательском тексте не используем.** Ни в интерфейсе, ни в подсказках, ни в письмах для людей. Вместо него: **и**, запятая, точка, скобки, по смыслу. (Дефис в словах вроде «по-русски» и обычный минус в числах остаются.)
5. В **документации для команды** в `docs/` технический тон допустим, но если копируешь фразу в UI, переписывай под правила выше.

## В чём разница (суть)

| Плохо | Хорошо |
|--------|--------|
| Обрывки с подписью «Что это.» / «Зачем тебе.» без связного смысла | Связный абзац или заголовок вроде «Зачем тебе это» и дальше полное объяснение |
| Две половинки, склеенные длинным тире: ответь на вопросы, потом покажем список | «Ответь на вопросы и в конце получишь подборку» (одна мысль целиком) |
| Внутренние пометки: демо, заглушки, вымышленные данные одной строкой | Простыми словами, отдельным предложением, что пока пример и ссылки не бронь |
| Фразы «без регистрации, без подбора людей» ради галочки | Только если человеку это реально важно; иначе убрать |
| «Учебный пример, не живая афиша» (обрыв второго сказуемого) | «Учебный пример, а не живая афиша» или две короткие фразы |

## Чего избегать в UI

- Формулировки из бэклога: «проверить гипотезу», «зафиксировать в рамках демо».
- Готовая строка «данные вымышленные, ссылки-заглушки» без перефраза для людей.
- Рубленый стиль ради краткости, если ломается чтение.
